# Annoy log

Append-only list of pipeline annoyances during phase work. Triaged at start of [Phase B](./ROADMAP.md#phase-b--extract-reusable-patterns) — pick at most 3 to fix.

**How to append:** date-stamped bullet, brief headline, then enough detail that future-you can decide if it's worth fixing without re-digging.

---

- **2026-05-23 — `attach_screens_on_approval` was silently failing.** Two bugs in `.docs/gate-watcher.py`: (1) section parser exited the "Screens delivered" block on `#### /login` / `#### /register` route subheadings (treated any `###*` line as a section boundary); (2) `_extract_url` didn't handle Linear's `[url](<url>)` auto-link form, so the captured URL contained `](<https://...>` and Linear rejected it with `INVALID_INPUT`. Both fixed. Underlying issue: `_gql` returns the raw response without checking the `errors` key, so all 18 mutation failures looked like successes — counter incremented, 0 actually persisted. Worth fixing `_gql` to raise on `errors`, but deferred (would change failure semantics for every mutation).

- **2026-05-23 — `pending-agents.json` accumulates stale + duplicate entries.** After today's three gate-watcher cycles, the queue had 16 entries: 4 dead `redesign-d-01-*` spawns from May 15–20 (D-01 has been approved + Done for hours), 9 `gate0-design-d-{01..09}` entries re-queued each time Gate 0 fires, plus `gate0-tl` / `gate0-qa` repeats. Only 1 (`fe-a-agent / dev-dev-01`) is actually live. Since the user has to manually pick entries from the Claude Code UI (scheduled-tasks blocked in unsupervised mode), every trigger is a "find the right one in a pile of garbage" task. Two real bugs underneath: (a) no completion-side cleanup — once a redesign is superseded or a design issue moves to Done, its queue entries should be pruned; (b) Gate-0 broadcast re-fires every cycle for every D-XX even when those entries already exist in the queue. Phase B candidate: add a `pending-agents.json` GC pass to gate-watcher (remove entries whose linked issue is Done, or whose `spawn_key` collides with an existing queue entry).

- **2026-05-23 — I-04 middleware allowlist excluded `/register`.** Same I-04 pass that shipped without root `middleware.ts` also wrote a public-route allowlist in `utils/supabase/middleware.ts` that only covered `/login` and `/auth`, leaving `/register` protected — so unauthenticated users (the only people who'd ever want to sign up) got bounced to `/login` instead. Hand-fixed inline. Reinforces the same Phase B candidate: a `pnpm build` + smoke-curl gate-check on infra issues (curl `/register` should return 200 for unauth) would have caught both this and the missing-middleware bug at I-04's done-transition.

- **2026-05-23 — I-04 (Supabase SSR auth) shipped without root `middleware.ts`.** TL agent created `utils/supabase/{client,server,middleware}.ts` but never the entry-point `middleware.ts` that Next.js actually loads, so the `updateSession` helper was unreachable and auth redirects didn't fire. I-04's Linear description does list "Create `middleware.ts`" but the gate-watcher had no acceptance check that the file exists. Hand-fixed inline (5-line wrapper re-exporting `updateSession`). Considered adding a "firefighter FE" agent for cases like this — decided against (scope gap, not role gap; CLAUDE.md forbids new agents in Phase A; the system is on probation per POST-MORTEM). Better fix at Phase B triage: add a `pnpm build` smoke check to gate-watcher's done-criteria so missing-glue-code can't slip past again.
